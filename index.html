Context Engineering

What this means & why it matters to the work of data engineering

What is Context Engineering?

When an AI agent runs a task, it needs more than a prompt. It needs the right context to make good decisions.

🗄️ Data - the actual tables, API responses, documents

🏷️ Semantics - what "customer" or "revenue" actually means across teams

🧠 Memory - what happened before: past decisions, corrections, exceptions

🔧 Tools - what the agent can do: run queries, call APIs, trigger workflows

🛡️ Guardrails - what the agent is not allowed to do: access rules, compliance, limits

Context is everything an agent needs beyond its training: Data gives it something to work with. Semantics ensures it understands correctly. Memory ensures it doesn't repeat mistakes. Tools let it take action. Guardrails ensure it doesn't cause harm.

As AI solutions scale, the audience for data engineering work has changed.

👤 Before

Humans only. We look at dashboards, interpret numbers, perform analytics, and ask for help when figures look off.

🤖 After

Includes AI agents. They read data and act on it directly. They don't stop to question whether the data makes sense.

Because agents won't question what they're given, it's on us to get the context right so that they can make the right decisions.

Building a strong data engineering core already supports this.

Most of what context engineering needs already exists in a mature data platform:

  • 📚 Data catalog → becomes the context layer agents query
  • 🔗 Lineage → tells agents where the data came from and if they can trust it
  • ✅ Quality monitoring → tells agents how reliable a source is right now
  • 📖 Business glossary → resolves ambiguity so agents use the right definition
  • 🔒 Access controls → governs what agents can and cannot touch

These are no longer nice-to-haves. These are the baseline every organisation needs.

But the baseline alone isn't enough.

To be future-ready, data engineering work should also include context engineering and shape how AI agents consume, interpret, and act on data at scale.

Data engineering

  • Build pipelines for human consumers
  • Freshness matters for dashboards
  • Definitions live in docs humans can read
  • Governance = who can access what

+ Context engineering

  • Build pipelines for AI agents too
  • Freshness directly affects agent decisions
  • Definitions must be machine-readable and unambiguous
  • Governance = what agents can access and do

Future-ready data engineering no longer just considers what works for human users, but also what works for AI agents.

For AI systems to scale fast and reliably, data engineering work should:

  • 🗄️ Drive a machine-readable metadata ecosystem: where to store it, how to scale it for agents
  • 📝 Evolve documentation to formalise important, unwritten rules that agents wouldn't discover by themselves
  • ⏱️ Make data freshness a top priority: outdated data means outdated decisions
  • 🔐 Extend access control to cover agent behavior, not just human access
  • 📊 Monitor context quality, not just data quality
  • 🤝 Data engineers need to be part of building AI systems

Data Engineering <> Context Engineering <> AI Systems

As AI solutions scale, the work of data engineering has evolved,
and we can be the first to recognise it, and own it.

With the right expertise and a commitment to a high-quality, context-rich data ecosystem,
we can enable AI agents to make reliable decisions at scale.
